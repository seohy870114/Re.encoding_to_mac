# 📓 Git Workflow & AI Agent Interaction Study

이 문서는 AI 에이전트와 협업하며 Git 작업을 진행할 때 발생할 수 있는 인지적 차이와 Git의 핵심 작동 원리를 정리한 공부 기록입니다. (노션 '바이브코딩스터디' 기록용)

## 1. 🔍 사건의 개요 (Context)
- **상황:** `app.py`와 `README.md`가 수정되었고, 새로운 파일(`IMPLEMENTATION_GUIDE.md`, `DEVELOPMENT_LOG.md`) 2개가 추가됨.
- **현상:** AI 에이전트가 신규 추가된 파일 2개만 `add/push` 하고, 기존에 수정된 `app.py`와 `README.md`는 누락함.
- **원인 분석:** 에이전트가 에디터의 '현재 화면(JSON context)' 정보에만 의존하여 작업을 수행했기 때문.

## 2. 💡 핵심 학습 내용 (Key Lessons)

### ① 정보 출처의 차이: Editor Context vs. Git Status
- **Editor Context (JSON):** "지금 사용자가 무엇을 보고 있는가?"를 알려주는 정보. (신규 파일이 열린 것 위주로 표시됨)
- **Git Status (CLI):** "실제 파일 시스템에서 무엇이 바뀌었는가?"를 알려주는 **최종 진실의 근거(Source of Truth)**.
- **교훈:** AI와 협업 시, 에디터 화면 정보만 믿지 말고 반드시 `git status`를 통해 실제 변경 내역을 '교차 검증'해야 함.

### ② Git의 'Selective Add' (선택적 추가) 원리
- `git add [파일명]`: 명시한 파일만 준비 영역(Staging Area)으로 보냄.
- `git add .`: 현재 디렉토리의 모든 변경 사항(수정, 삭제, 추가)을 한꺼번에 보냄.
- **분석:** 에이전트가 특정 파일명만 지정해 `add`했기 때문에, Git은 철저하게 명령받은 파일만 처리하고 나머지는 'Unstaged' 상태로 남겨둠. 이는 Git의 강력한 통제권이자 실수의 원인이 될 수 있음.

### ③ AI 에이전트의 인지 흐름 (Mental Model)
1. **입력:** "새 파일이 열렸습니다"라는 정보 수신.
2. **판단:** "사용자가 이 파일을 올리고 싶어 하는구나"라고 단정함.
3. **실수:** 전체 상태(`git status`)를 확인하지 않고 눈에 보이는 것만 처리함.
4. **교정:** 사용자의 지적 이후 `git status`를 통해 숨겨진 수정 사항을 발견하고 처리함.

## 3. 🛠 향후 재발 방지 전략
- **에이전트 측:** 어떤 Git 작업을 하기 전에도 반드시 `git status`를 먼저 실행하여 전체 변경 사항을 리스트업할 것.
- **사용자 측:** "바뀐 것 다 올려줘"와 같이 명시적으로 전체 처리를 요구하거나, 에이전트가 `status`를 확인했는지 체크할 것.
